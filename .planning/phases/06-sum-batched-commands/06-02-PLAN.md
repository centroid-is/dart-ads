Sum handler logic (per 06-RESEARCH §Mock Dispatch Design):
- `N = offset` (outer indexOffset carries the item count). VALIDATE: for 0xF080/0xF081 require `writeLength == N*12` (plus, for WRITE, the sum of per-item lengths); for 0xF082 require the 16B stride is consistent. On any mismatch or out-of-bounds walk, `break` with NO response (mirrors existing hostile-input discipline).
- Loop i in 0..N, parsing each item header from the write-buffer at the correct stride (12B for READ/WRITE, 16B for READWRITE). Per-item error injection: if the item's inner `indexGroup == kErrResultGroup` (0xE7700000), that item's result word = its inner `indexOffset` and it contributes ZERO data bytes (READ/READWRITE). Otherwise result = 0 and the item hits `store`: READ copies min(len, stored) padded to len; WRITE stores the item's write-data slice (write-back); READWRITE stores write payload then reads back at min(readLen, stored) — the RETURNED length equals the bytes actually produced.
- Assemble the response per the pinned layout: 0xF080 → N×u32 errs THEN concatenated successful-item data at requested lengths; 0xF081 → N×u32 errs only; 0xF082 → N×(err u32, retLen u32) THEN concatenated data at retLen. Cap total sizes against `kMaxFrameBytes` before allocating (T-6-02). Wrap with `wrapResponse(f, READ_WRITE, ...)`, set `haveRes = true`.

Add a code comment stating the failed-item = 0-data-bytes convention is the frozen contract shared with the Dart decoder and goldens (Phase 9 audit note).
