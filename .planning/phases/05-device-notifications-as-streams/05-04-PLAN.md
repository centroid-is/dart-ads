    - addNotification(payload, ctrl) sends cmd 0x06, and on a success response registers
      `_demuxControllers[handle] = ctrl` BEFORE the returned Future completes, then returns the handle.
    - A 0x08 frame for that handle arriving in the SAME inbound chunk as the Add-response is delivered
      to ctrl (not dropped) — the race test: inject Add-response + notification in one FakeTransport
      chunk and assert ctrl received the sample.
    - A non-zero AMS errorCode or non-zero decoded result throws AdsException (surfaced to the caller),
      and no controller is registered.
    - deleteNotification(handle) sends cmd 0x07, removes the handle from _demuxControllers, and closes
      the controller; a dead connection makes it throw (the client swallows — see 05-05), it does NOT
      throw here spuriously.
  </behavior>
  <action>
    Retype `_demuxControllers` to `Map<int, StreamController<AdsNotification>>`. Add
    `Future<int> addNotification(Uint8List payload, StreamController<AdsNotification> ctrl,
    {Duration? timeout})`: `await request(AdsCommandId.addDeviceNotification, payload, ...)`, throw
    `AdsException.fromCode(errorCode)` if the AMS errorCode is non-zero, `decodeAddNotificationResponse`
    the payload, throw on non-zero `result`, then `_demuxControllers[handle] = ctrl` and return the
    handle. CRITICAL: the map write happens in the same async continuation that first observes the
    response, and BEFORE any 0x08 for that handle can be dispatched — because the peer cannot send a
    notification for handle H until it has sent H's Add-response, and both travel in-order on one TCP
    connection, so registering here (option A) is sufficient; do not add a holding buffer. Add
    `Future<void> deleteNotification(int handle, Uint8List payload, {Duration? timeout})`:
    `await request(AdsCommandId.deleteDeviceNotification, payload, ...)`, then
    `_demuxControllers.remove(handle)?.close()`. Add a public `int droppedNotifications = 0;` field

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PLC/peer -> _onFrame 0x08 branch | Untrusted notification frames cross into the demux dispatch |
| Reconnected session -> stale handle | An old handle number may belong to a different subscription after reconnect |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-02 | DoS | Malformed 0x08 frame reaching the connect() listener | mitigate | Local try/catch in the 0x08 branch; `droppedNotifications++`; never rethrow so `_failClose` is not tripped — one hostile frame cannot kill other subscriptions |
| T-5-03 | Tampering | Per-sample size/handle lies | mitigate | parseNotificationStream (05-01) bounds-checks; unknown handle -> silent ignore (C++ parity), no cross-dispatch |
| T-5-10 | Spoofing | Deleting a stale handle on a new session | mitigate | `_failClose` invalidates+clears the map locally on disconnect; no Delete is issued against a reconnected session (enforced at the client in 05-05) |
| T-5-11 | DoS | First notification dropped (race) | mitigate | Synchronous demux registration in addNotification (option A) before the Future completes |
</threat_model>
