<objective>
Implement the pull/push snapshot-and-apply pair (CLI-05, CLI-06): `pull` browses
the symbol table and (with `--values`) sum-reads every value into a JSON file;
`push` reads that file back and sum-writes the values with `--dry-run` preview and
a per-item pass/fail report. The pull->push round-trip must be lossless — the
project's headline pull/push contract.

Purpose: pull/push is the batched, whole-PLC verb pair proving sum-read/sum-write
end-to-end and the lossless snapshot contract. Output: two command files + one
integration test. The JSON schema (document it in the pull file header) is Claude's
discretion per 08-CONTEXT.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| snapshot file -> push | file contents are untrusted (hand-edited, possibly hostile JSON) |
| terminal flags -> pull/push | operator flags untrusted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-02 | Tampering | push snapshot-file parsing | mitigate | jsonDecode + schema/shape validation guarded -> FormatException (exit 2); each value parseHex-bounded to the declared size; no unbounded allocation from a hostile count/length |
| T-8-08 | Tampering | wrong values pushed to PLC | mitigate | --dry-run preview before any write; per-item report; exit 1 on any failed item so failures are never silent |
| T-8-09 | Denial | huge symbols array in a hostile file | mitigate | sumWrite is one bounded round-trip; cap item count sanity before batching (document the ceiling) |
| T-8-SC | Tampering | pub installs | accept | no new packages beyond 08-01's `args` (Approved) |
</threat_model>
